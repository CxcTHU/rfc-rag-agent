# 阶段 64 - 开发记录

> 状态：已完成，用户人工功能验收 PASS
> 阶段总览：[[阶段 64 - 主流 Agent 延迟优化]]
> 收尾交接：[[阶段 64 - 收尾交接]]

## 目标

把 Agent 的 planner、检索/rerank、首个真实回答 token 与最终完成时间分段观测，
在不关闭官方 GLM rerank、不固定 Dynamic-K、不使用语义答案缓存的前提下，降低
用户感知延迟。

## 开发主线

1. 建立冻结交替 A/B 合同与安全指标，只保存 case id、类别、数值耗时、计数和配置标签。
2. 实现 B-only Route-First、Harness short loop、复杂请求有界 fan-out 与安全 trace。
3. 优化本地计算：有界图遍历、BM25 候选预筛和启动预热；保持原评分语义。
4. 将默认实际 reranker 迁移到官方 `zhipu/rerank`；75 保持为候选池，最终来源仍是
   Dynamic-K 4–12。
5. 修复人工核验发现的同会话第二轮计时卡住和图片证据未内联展示。

## 关键变更

- `app/services/agent/route_first.py`：快/复杂路径判定及一次升级边界。
- `app/services/retrieval/fanout_limiter.py`：复杂检索并发上限。
- `app/services/graphrag/`：不复制全图的有界遍历与延迟诊断。
- `app/services/retrieval/bm25_search.py`：等价评分前的候选预筛、启动预热。
- `app/services/generation/http_pool.py`：完整消费 SSE 后的连接复用。
- `frontend/src/features/chat/`：终态 run 的新请求接管与图片证据卡片。

## 验证与效果

| 项目 | 优化前/对照 | 优化后 | 结论 |
|---|---:|---:|---|
| 图检索本地 profile | 约 12.6 秒 | 约 0.31 秒 | 有界遍历显著降耗 |
| BM25 warm profile | 约 4.3 秒 | 约 1.9 秒 | 评分/排序语义保持 |
| 官方 rerank P95（5 次、12 candidates） | 旧云 2141.696 ms | 官方 378.450 ms | 方向性稳定性改善 |
| B Flash 首 token P95（3 对） | A Pro 39.831 秒 | B 18.683 秒 | 更快，但未达 15 秒目标 |

最终自动化：后端 `1479 passed, 1 skipped`；前端单测 31 项、lint、build 通过；
Stage 30 为 `91.52 / A / pass`。

## 诚实结论

功能验收 PASS 不等于原性能门禁 PASS。完整 30×3 冻结冷链路 A/B 和盲评未运行，
因此首 token P95 ≤15 秒仍是下一阶段验证项。
