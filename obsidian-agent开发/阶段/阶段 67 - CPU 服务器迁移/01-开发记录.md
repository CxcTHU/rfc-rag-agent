# 阶段 67 - 开发记录

## 运行步骤持久化修复

根因是实时 SSE 事件只存在前端 `message.events` 内存；完成响应的 `workflow_steps` 只有最终检索与回答记录。30 秒 stale 后发生 refetch、页面刷新或重新登录时，数据库无法恢复实时事件，界面因此缩减。

修复链路：

1. stream adapter 收集最后 64 条已经白名单过滤的 runtime event；
2. 以独立 `runtime_workflow_steps` 写入 response 和 assistant metadata；
3. 会话 hydration 明确恢复该数组；
4. 完成态优先使用完整运行步骤，旧数据继续回退 `workflow_steps` / `tool_calls`；
5. 工具 start/result 在前端保持既有生命周期合并，不伪造 latency trace 步骤。

E2E 先红后绿：旧 mock 合同刷新前 3 步、刷新后 2 步；更新后 3-case Chromium 全部通过，比较恢复前后步骤数量与展开标签序列。评测只使用合成短文本和安全摘要。
