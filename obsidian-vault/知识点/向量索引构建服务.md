# 向量索引构建服务

所属阶段：[[阶段 2 - Embedding 与向量检索]]
所属分类：[[RAG 链路]]
相关位置：`app/services/retrieval/vector_index.py`、`scripts/build_vector_index.py`

## 它解决什么问题

用户搜索时不应该临时为全库 chunks 生成 embedding。向量索引应该提前构建，搜索时只需要把用户问题向量化并比较相似度。`VectorIndexService` 就是负责提前构建和维护索引的服务。

## 在本项目中怎么用

`VectorIndexService.build_index()` 会扫描 chunks，计算内容 hash，判断已有 embedding 是否需要跳过、插入或更新。命令行入口是 `scripts/build_vector_index.py`。

## 新词解释

- index：索引，为了更快检索而提前准备的数据。
- stale embedding：过期向量，表示 chunk 内容变了但旧向量还没更新。
- batch：批次，把多个 chunk 分组处理。
- batch commit：批量提交数据库写入，减少大量写入时的磁盘提交次数。
- idempotent：幂等，重复运行不会生成重复数据。

## 为什么这样设计

索引构建和查询检索是两个不同职责。把构建逻辑独立出来，可以让 API、脚本和后续后台任务复用同一个服务。阶段 2 首次评测时发现逐条提交较慢，因此改成按 batch 提交。

## 面试可能怎么问

问：如何保证索引脚本可以重复运行？

答：每条 embedding 都记录 content_hash 和唯一约束。服务会比较当前 chunk 内容和已有 embedding 的 hash，没变就跳过，变了就更新，没索引就插入，所以重复运行是安全的。

## 你应该怎么回答

我把向量索引构建做成 `VectorIndexService`，而不是写在 API 里。它支持跳过未变化的 chunk，更新过期 embedding，并按 batch 提交数据库写入。这样索引脚本可以重复运行，也为后续后台任务或向量库迁移打基础。
