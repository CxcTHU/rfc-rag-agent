# 健康检查接口

所属阶段：[[阶段 0 - FastAPI 工程底座]]
所属分类：[[工程化与可观测性]]
相关位置：`app/api/health.py`

## 它解决什么问题

健康检查接口用于回答一个最基础的问题：服务是否活着，是否能被外部访问。

## 在本项目中怎么用

本项目实现了：

```text
GET /health
```

返回：

```json
{
  "status": "ok",
  "service": "RFC-RAG-Agent",
  "environment": "development"
}
```

## 为什么这样设计

阶段 0 的健康检查只检查 API 服务本身。后续可以扩展为检查：

- SQLite 是否可连接。
- 向量库是否可用。
- embedding 模型配置是否存在。
- 聊天模型服务是否可访问。

## 面试可能怎么问

`/health` 这么简单，有什么工程意义？

## 你应该怎么回答

健康检查是服务可观测性的起点。开发阶段它能证明服务能启动；部署阶段它可以给负载均衡、容器平台或监控系统使用。后续 RAG 系统依赖数据库、向量库和模型服务，健康检查可以逐步暴露这些依赖的状态。
