# 阶段 3 - 引用式问答

所属索引：[[阶段索引]]

## 阶段目标

把检索召回的 chunk 组织成上下文，调用国产聊天模型生成回答，并返回引用来源。资料不足时明确拒答。

## 完成内容

- 新增 `ChatModelProvider`，支持 deterministic provider 和 OpenAI-compatible provider。
- 新增 RAG prompt/context builder，把检索结果组织成带 `[1]`、`[2]` 编号的上下文。
- 新增 `CitationAnswerService`，串联检索、prompt、模型调用、引用提取和拒答。
- 新增 `POST /chat`，返回 answer、citations、sources、refused、retrieval_mode 和 model 信息。
- 新增 `qa_logs` 问答日志，记录问题、答案、召回 chunk、引用、模型和拒答状态。
- 新增 chat 评测集和 `scripts/evaluate_chat.py`。

## 面试表达

阶段 3 我完成了引用式问答的最小稳定链路。我先抽象 `ChatModelProvider`，让业务逻辑不绑定某一家模型服务；再用 prompt builder 把检索结果组织成带来源编号的上下文；`CitationAnswerService` 负责检索、prompt 构造、模型调用、引用提取和拒答判断；最后通过 `POST /chat` 返回答案、引用、来源、拒答状态和模型信息。

为了保证可追溯，我让 citations 只能引用本次 sources 中存在的编号，并新增 `qa_logs` 记录问答日志。为了避免只靠演示判断效果，我新增 chat 评测集和 `evaluate_chat.py`，当前 chat 评测 6/6 通过。
